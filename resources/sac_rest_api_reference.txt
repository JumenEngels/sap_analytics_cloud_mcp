################################################################################
PART 1: GETTING STARTED
################################################################################

The SAP Analytics Cloud REST API allows authorised third-party applications to
programmatically access data stored on an SAC tenant.  The API conforms to the
OpenAPI specification and uses OAuth 2.0 for authentication.

Base URL pattern:
  https://<tenant>.<dataCenter>.sapanalytics.cloud

All API endpoints are appended to this base URL (e.g. /api/v1/stories).

Endpoint overview (from official Getting Started):
  api/v1/stories         Allows you to get, copy, rename, and delete stories.
  api/v1/$metadata       Returns OData metadata document for the /api/v1 service.
  api/v1/Resources       Allows you to view resources (stories, models, folders).
  api/v1/filerepository  Lists files in the file repository (widgets, images, CSS).
  api/v1/Repositories    Lists custom widget repositories.
  api/v1/widgetquery     Queries widget data programmatically.
  api/v1/dataexport      OData-compliant pull access to model data.
  api/v1/dataimport      Job-based import of fact/master/currency data.
  api/v1/multiActions    Trigger automated planning sequences via API.
  api/v1/calendar        Manage planning calendar events (tasks, processes).
  api/v1/content         Import/export SAC artefacts between tenants.
  api/v1/scim            Manage users and teams (SCIM v1, v2, v3).
  api/v1/audit           Export audit log activities.
  api/v1/monitoring      System health and monitoring data.
  api/v1/ps              Scheduled publications (automated story distribution).
  api/v1/ts              Translation of SAC content via XLIFF.
  api/v1/csrf            Fetch CSRF token for write operations.

--------------------------------------------------------------------------------
1.1 Setting Up Authentication (OAuth 2.0)
--------------------------------------------------------------------------------

SAC supports four authentication methods (per official SAP documentation):

A) OAuth 2.0 SAML Bearer Assertion (2-legged)
   - Use case: apps deployed on SAP BTP using the BTP SDK.
   - Requires a SAML assertion signed by a Trusted Identity Provider (IdP).
   - The SAML assertion is exchanged for a Bearer access token.
   - Typically used in BTP subaccount-to-SAC integration scenarios.

B) OAuth 2.0 Client Credentials Grant (2-legged)
   - Use case: non-browser, technical-user access to the REST API.
   - Server-to-server, no user interaction required.
   - Register in SAC: Main Menu > System > Administration > App Integration
     > +Add New OAuth Client, select "Client Credentials".
   - Requires a SAML Bearer Assertion signed by a trusted IdP.
   - Save the IdP certificate under Trusted Identity Providers in SAC Admin.
   - Flow:
     1. Client constructs SAML assertion (or uses BTP built-in support)
     2. POST to Token URL with client_id, client_secret, SAML assertion
     3. Receive access_token (Bearer)

C) OAuth 2.0 Authorization Code Grant (3-legged)
   - Use case: non-browser apps acting on behalf of a named user.
   - Register in SAC: Main Menu > System > Administration > App Integration
     > +Add New OAuth Client, select "Authorization Code".
   - Configure: Name, ID, Secret (optional), Redirect URI, token lifetimes.
   - Flow:
     1. Redirect user to Authorization URL (found in App Integration tab)
     2. User authenticates with the Identity Provider (IdP)
     3. IdP returns authorization code to Redirect URI
     4. Client exchanges code for access token via POST to Token URL

D) Basic Authentication Against the Token Service  ← simplest, recommended
   - Use case: simplest option for dev/automation and SCIM provisioning.
   - Register an OAuth client in SAC Admin > App Integration.
   - Flow:
     1. POST to Token URL with URL parameter ?grant_type=client_credentials
     2. Include header: Authorization: Basic Base64(<ClientID>:<Secret>)
     3. Response: { "access_token": "...", "token_type": "bearer",
                     "expires_in": 3600 }
     4. Use the access_token as Bearer token on all subsequent API calls.

   Example token request:
     POST <Token URL>?grant_type=client_credentials
     Headers:
       Authorization: Basic dXNlcjpwYXNz    (Base64 of ClientID:Secret)
       Content-Type: application/x-www-form-urlencoded

Key URLs (found in SAC Admin > App Integration > OAuth Clients):
  - Authorization URL  (for Auth Code flow)
  - Token URL          (for all flows — token exchange endpoint)

For BTP-hosted SAC tenants (newer deployments):
  - OAuth is configured via BTP service keys (XSUAA).
  - token_url typically:
    https://<subdomain>.authentication.<region>.hana.ondemand.com/oauth/token
  - grant_type: client_credentials
  - client_id / client_secret come from the service key JSON.

Session Management (required for all methods):
  - A cookie store MUST be maintained across all API calls.
  - x-sap-sac-custom-auth: true is required on every API request.
  - Write ops (POST/PUT/PATCH/DELETE) require a CSRF token (see 1.2).
  - The CSRF token and session cookies must travel together.

--------------------------------------------------------------------------------
1.2 Get a Valid CSRF Token
--------------------------------------------------------------------------------

CSRF tokens are required for write operations (POST, PATCH, PUT, DELETE).
Read-only GET requests need only the Bearer token.

Procedure:
  1. Make a GET request to any API endpoint (e.g. /api/v1/csrf).
  2. Add these headers:
       Authorization: Bearer <access_token>
       x-csrf-token: Fetch
       x-sap-sac-custom-auth: true
  3. Read the response headers:
       x-csrf-token: <tokenValue>
       Set-Cookie: <session cookies>
  4. Include both the x-csrf-token and the cookies in subsequent write requests.

Note: CSRF tokens are session-scoped; re-fetch if the session expires.

--------------------------------------------------------------------------------
2.1 api/v1/stories
--------------------------------------------------------------------------------

Allows you to get, copy, rename, and delete stories available on your SAP
Analytics Cloud tenant.

Prerequisites:
  - SAP Analytics Cloud tenant URL and user account.
  - OAuth configured (see Section 1.1).
  - Valid CSRF token for POST, PUT, PATCH, DELETE requests (see Section 1.2).

. . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . .

GET /api/v1/stories

  Returns a list of stories and story metadata accessible to the specified user.

  Three object types may be returned: stories, models, and remote connections.

  Only models based on remote data sources have remote connection information.
  All remote connections will have id, connectionType, and systemType attributes.
  Different remote connections can have different sets of attributes based on
  the connection type and the system type.  Only the attributes applicable to
  the remote system type the model is stored on will appear in the response.

  Request headers:
    Authorization: Bearer <access_token>
    (Optional) x-csrf-token: <fetch>
    (Optional) x-sap-sac-custom-auth: true
      Add this if you are using basic authentication against the token service.

  URL parameters:
    (Optional) include=models
      The metadata of the models used in each story will be returned.
      Example: api/v1/stories?include=models

  Response (200 OK):
    An array of story, model, or remote connection objects.

  Response example:
    [
      {
        "id": "8114AB5AE4CA9906E10000000A4E7468",
        "name": "My Story",
        "description": "",
        "created": "2018-03-16T14:43:18.081Z",
        "createdBy": "admin",
        "changed": "2018-04-13T06:54:28.412Z",
        "changedBy": "admin",
        "models": [
          { "id": "t.X.Model:Model", "description": "My Model" }
        ],
        "openURL": "/sap/fpa/ui/tenants/000/bo/story/<STORY_ID>",
        "isTemplate": false,
        "isSample": false
      }
    ]

. . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . .

POST /api/v1/stories?copyFrom=<SOURCE_STORY_ID>

  Copies the story specified by <SOURCE_STORY_ID>.  By default, the copied
  story will be created in its source folder.  The response body contains
  the new story name.

  Request headers:
    Authorization: Bearer <access_token>
    x-csrf-token: <csrf_token>
    x-sap-sac-custom-auth: true
    (Optional) x-sap-sac-custom-auth: true
      Add this if you are using basic authentication against the token service.

  URL parameters:
    copyFrom (required): Specifies the story to copy.
      Example: api/v1/stories?copyFrom=<SOURCE_STORY_ID>

    copyTo (optional): Specifies the folder where the copied story will be
      created.  If not specified, the copy is created in the same folder as
      the original.
      Example: api/v1/stories?copyFrom=<SOURCE_STORY_ID>&copyTo=<FOLDER>

  Request body (optional):
    {
      "name": "<NEW_STORY_NAME>"
    }
    If a name is not specified, "Copy" will be added to the original story name.

  Response (201 Created):
    Returns the source story ID and the copied story ID.

. . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . .

PATCH /api/v1/stories/<STORY_ID>

  Renames the story specified by <STORY_ID>.

  Request headers:
    Authorization: Bearer <access_token>
    x-csrf-token: <csrf_token>
    x-sap-sac-custom-auth: true
    (Optional) x-sap-sac-custom-auth: true
      Add this if you are using basic authentication against the token service.

  URL parameters:
    <STORY_ID>: The ID of the story to rename.
      Example: api/v1/stories/<STORY_ID>

  Request body (required):
    {
      "name": "<NEW_STORY_NAME>"
    }
    Note: "name" must be included in the request body.

  Response (200 OK):
    The response contains the new story name.

. . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . .

DELETE /api/v1/stories/<STORY_ID>

  Deletes the story specified by <STORY_ID>.

  Request headers:
    Authorization: Bearer <access_token>
    x-csrf-token: <csrf_token>
    x-sap-sac-custom-auth: true
    (Optional) x-sap-sac-custom-auth: true
      Add this if you are using basic authentication against the token service.

  URL parameters:
    <STORY_ID>: The ID of the story to delete.
      Example: api/v1/stories/<STORY_ID>

  Response (200 OK):
    Successful operation.

. . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . . .

Error Responses (Stories API):

  HTTP Code   Reason
  --------    ---------------------------------------------------------------
  400         Bad request related to nonexistent property.
              There is a typo or mistake in your request.  Check that the
              request syntax is correct.

  403         User is not permitted to perform the requested operation.
              This may occur if the user is authenticated by the IdP but
              does not have a corresponding user in the SAC tenant.
              Ensure that a user exists in SAP Analytics Cloud.

  404         Resource not found.
              The specified story ID could not be found.  Ensure the ID is
              correct and the user has access rights to the specified story.

  409         Conflict with existing resource.
              Occurs while making a PATCH request if the specified new story
              name already exists.

  500         A server error has occurred.
              The response body will be a JSON object with status, code, and
              message properties that may help determine the cause.

2.3 api/v1/Resources
--------------------------------------------------------------------------------

Endpoint: GET /api/v1/Resources

OData-compliant endpoint.  Lists resources (stories, models, etc.) stored on
the tenant.

Supports standard OData query options:
  $filter    e.g. $filter=type eq 'STORY'
  $select    e.g. $select=id,name,type
  $top       e.g. $top=10
  $skip      e.g. $skip=20
  $orderby   e.g. $orderby=name asc
  $count     returns inline count

Single resource by ID:
  GET /api/v1/Resources('<resourceId>')

Resource types include: STORY, MODEL, FOLDER, etc.

--------------------------------------------------------------------------------
2.4 api/v1/filerepository
--------------------------------------------------------------------------------

Endpoint: GET /api/v1/filerepository/Resources

Lists files in the file repository (custom widgets, images, CSS, etc.).
Supports the same OData query options as above.

Single resource:
  GET /api/v1/filerepository/Resources('<resourceId>')

Metadata:
  GET /api/v1/filerepository/$metadata

Pagination: Results come in batches of 50.  If more exist, the response
includes "@odata.nextLink" with the URL for the next page.

2.5 Additional Query Options (Content APIs)

  $expand=<navProperty>    Expand navigation properties inline.
  $top=<n>                 Limit results.
  $skip=<n>                Skip first n results.
  $orderby=<prop> asc|desc Sort results.
  $count=true              Include total count in response.

################################################################################
PART 3: EXPORTING DATA (Data Export Service / DES)
################################################################################

The Data Export Service (DES) is a fully OData-compliant REST API providing
on-demand "pull" access to model data.

Base URL:
  https://<tenant>.<dataCenter>.sapanalytics.cloud/api/v1/dataexport

All DES endpoints use GET (not POST).

--------------------------------------------------------------------------------
3.1 Authentication for DES
--------------------------------------------------------------------------------

DES uses a separate OAuth client configuration (can be the same or separate):
  - grant_type: client_credentials (via BTP XSUAA or SAC OAuth client)
  - Scope: may require specific DES scopes depending on tenant config.

CSRF tokens: same procedure as Section 1.2 but fetched from:
  GET /api/v1/dataexport/administration/$metadata
  with header: x-csrf-token: Fetch

--------------------------------------------------------------------------------
3.2 Data Export API Changelog
--------------------------------------------------------------------------------

Check the official SAP release notes for your tenant's quarterly release.
The DES API evolves with new entity sets and query options each quarter.

3.32 Additional Query Options for Data Export
--------------------------------------------------------------------------------

All /providers/ endpoints support standard OData v4 query options:
  $select=<properties>       Select specific columns.
  $filter=<expression>       e.g. $filter=Country eq 'US'
  $top=<n>                   Limit rows.
  $skip=<n>                  Skip rows.
  $orderby=<prop> asc|desc   Sort.
  $count=true                Include total count.
  $format=json|xml|csv       Response format.
  $expand=<navProp>          Expand navigation properties.

SAC-specific:
  pagesize=<n>               Controls page size (max compressed response 30MB).

--------------------------------------------------------------------------------
3.33 Pagination of Exported Data
--------------------------------------------------------------------------------

If data exceeds one page, the response includes:
  "@odata.nextLink": "https://...?$skiptoken=..."

Follow the nextLink URL to retrieve the next page.
Maximum page size: 30MB compressed.

Continue following nextLinks until no more are returned.

--------------------------------------------------------------------------------
3.34 Error Codes & Troubleshooting (Data Export)
--------------------------------------------------------------------------------

Common HTTP status codes:
  200 OK                   Success.
  400 Bad Request          Malformed query / invalid parameters.
  401 Unauthorized         Invalid or expired bearer token.
  403 Forbidden            Insufficient permissions / missing CSRF token.
  404 Not Found            Namespace, Provider, or entity set not found.
  429 Too Many Requests    Rate limit exceeded.
  500 Internal Server Error Server-side issue.

Common error scenarios:
  - "CSRF token validation failed" → Re-fetch CSRF token.
  - "Provider not found" → Verify ProviderID via /Namespaces('sac')/Providers.
  - "Unauthorized" → Check OAuth token expiry, re-authenticate.


################################################################################
PART 4: IMPORTING DATA (Data Import Service / DIS)
################################################################################

Base URL:
  https://<tenant>.<dataCenter>.sapanalytics.cloud/api/v1/dataimport

The Data Import Service provides a step-by-step job-based approach for
importing fact data, master data, and currency rates into New Models.

Note: Only works with New Models (not Classic Account Models).

--------------------------------------------------------------------------------
4.1 Import Job Workflow
--------------------------------------------------------------------------------

Step 1: Create a job
  POST /api/v1/dataimport/models/{modelID}/factData
  POST /api/v1/dataimport/models/{modelID}/masterData
  POST /api/v1/dataimport/models/{modelID}/currencyRates

  Response: { "jobID": "<jobID>" }

Step 2: Upload data to staging area
  POST /api/v1/dataimport/jobs/{jobID}
  Content-Type: text/csv or application/json
  Body: CSV or JSON payload with the data rows.

  Can be called multiple times to append data in chunks.

Step 3: Validate the job
  POST /api/v1/dataimport/jobs/{jobID}/validate

  Response: validation result with warnings/errors.

Step 4: Run the import
  POST /api/v1/dataimport/jobs/{jobID}/run

  Response: { "status": "RUNNING" }

Step 5: Check status
  GET /api/v1/dataimport/jobs/{jobID}/status

  Response: { "status": "COMPLETED" | "FAILED" | "RUNNING", ... }

All POST operations require CSRF token + Bearer token.

--------------------------------------------------------------------------------
4.2 Import Endpoints Summary
--------------------------------------------------------------------------------

POST /api/v1/dataimport/models/{modelID}/factData         Create fact job
POST /api/v1/dataimport/models/{modelID}/masterData        Create master job
POST /api/v1/dataimport/models/{modelID}/currencyRates     Create rates job
POST /api/v1/dataimport/jobs/{jobID}                       Upload data
POST /api/v1/dataimport/jobs/{jobID}/validate              Validate
POST /api/v1/dataimport/jobs/{jobID}/run                   Execute

PART 8: MANAGING USERS AND TEAMS (SCIM APIs)
################################################################################

SAC provides three versions of the SCIM API:

================================================================================
8.1 api/v1/scim (v1)
================================================================================

Base: /api/v1/scim

Endpoints:
  GET    /scim/Users               List all users.
  POST   /scim/Users               Create a user.
  GET    /scim/Users/<ID>           Get user by ID.
  PUT    /scim/Users/<ID>           Replace user.
  PATCH  /scim/Users/<ID>          Update user (partial).
  DELETE /scim/Users/<ID>           Delete user.
  POST   /scim/Users/.search       Search users with filter body.

  GET    /scim/Groups              List all teams/groups.
  POST   /scim/Groups              Create a team.
  GET    /scim/Groups/<Team_ID>    Get team by ID.
  PUT    /scim/Groups/<Team_ID>    Replace team.
  PATCH  /scim/Groups/<Team_ID>    Update team (partial).
  DELETE /scim/Groups/<Team_ID>    Delete team.

User attributes (key fields):
  id, userName, name.givenName, name.familyName, emails[].value,
  displayName, active, roles[], preferredLanguage, locale, timezone

Team attributes:
  id, displayName, members[].value (user IDs), members[].$ref

================================================================================
8.2 api/v1/scim2 (v2 — recommended)
================================================================================

Base: /api/v1/scim2

SCIM 2.0 compliant (RFC 7643, 7644).

Endpoints:
  POST   /scim2/Bulk                       Bulk operations.
  GET    /scim2/Users                       List users.
  POST   /scim2/Users                       Create user.
  GET    /scim2/Users/<UUID>                Get user by UUID.
  PUT    /scim2/Users/<UUID>                Replace user.
  PATCH  /scim2/Users/<UUID>                Update user.
  DELETE /scim2/Users/<UUID>                Delete user.
  POST   /scim2/Users/.search               Search users.

  GET    /scim2/Groups                      List groups/teams.
  POST   /scim2/Groups                      Create group.
  GET    /scim2/Groups/<UUID>               Get group.
  PUT    /scim2/Groups/<UUID>               Replace group.
  PATCH  /scim2/Groups/<UUID>               Update group.
  DELETE /scim2/Groups/<UUID>               Delete group.
  POST   /scim2/Groups/.search              Search groups.

  GET    /scim2/ServiceProviderConfig       SCIM service config.
  GET    /scim2/Schemas                     Available schemas.
  GET    /scim2/ResourceTypes               Resource type definitions.
  GET    /scim2/ResourceTypes/<Type>        Single resource type.

Search request body (POST .search):
  {
    "schemas": ["urn:ietf:params:scim:api:messages:2.0:SearchRequest"],
    "filter": "displayName co \"admin\"",
    "startIndex": 1,
    "count": 25
  }

Bulk request body (POST /Bulk):
  {
    "schemas": ["urn:ietf:params:scim:api:messages:2.0:BulkRequest"],
    "Operations": [
      {
        "method": "POST",
        "path": "/Users",
        "data": { "userName": "newuser", ... }
      }
    ]
  }

================================================================================
8.3 api/v1/scim3 (v3 — groups only)
================================================================================

Base: /api/v1/scim3

Extended group management:

  GET    /scim3/Groups              List groups.
  POST   /scim3/Groups              Create group.
  GET    /scim3/Groups/<UUID>       Get group.
  PUT    /scim3/Groups/<UUID>       Replace group.
  PATCH  /scim3/Groups/<UUID>       Update group.
  DELETE /scim3/Groups/<UUID>       Delete group.
  POST   /scim3/Groups/.search      Search groups.

================================================================================
8.4 Optional User Parameters
================================================================================

When creating/updating users, optional parameters include:
  preferredLanguage, locale, timezone, active (boolean),
  roles (array of role objects), manager, department, title,
  phoneNumbers, photos, addresses, customAttributes

================================================================================
8.5 Assigning Roles to Users and Groups
================================================================================

Roles are assigned via the "roles" attribute in user objects, or by adding
users to role-associated groups.

Example role assignment in user PATCH:
  {
    "Operations": [
      {
        "op": "add",
        "path": "roles",
        "value": [{ "value": "ROLE_ID", "display": "Admin" }]
      }
    ]
  }
